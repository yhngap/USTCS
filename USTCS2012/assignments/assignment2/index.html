  • Polymorphism

Overview

There are 5 classes.

  • Transaction: is an abstract base class. This is the basic unit of record in ledger.
  • Income: is a type of transaction which is implemented slightly differently on top of transaction.
  • Expenses: is a type of transaction which is implemented slightly differently on top of transaction.
  • Ledger: keeps all the transactions
  • Account: keeps track of different ledgers and different kinds of balances.

    • The relations of the classes are illustrated with the following UML diagram. You may refer to this wiki page to understand more about the notations used in the UML diagram.

Your solution to this assignment must be based on the given skeleton code provided in the Download section.

Your task is to complete the following:

  • Account implementation in account.cpp
  • Expenses implementation in expenses.cpp
  • Income implementation in income.cpp
  • Ledger implementation in ledger.cpp
  • Transaction implementation in transactions.cpp

FAQ

Frequently Asked Questions


Q: My code doesn't work / there is an error, here is the code, can you help me fix it?

A: As the assignment is a major course assessment, to be fair, you are supposed to work on it on your own and we should not finish the tasks for you. We might provide some very general hints to you, but we shall not fix the problem or debug for you.


Q: Can I add extra helper functions?

A: You may do so in the files that you are allowed to modify and submit.



Q: Can I use global variable or static variable such as "static int x"?

A: No.


Q: Can I use "auto"?

A: No.



Q: Regarding the description, it asks us to "modify" the 5 cpp files but indeed I could only find 2. Are we supposed to create the remaining 3 by our own?

A: Yes. Please ensure you have created/modified all the cpp required.



Q: For some functions we are not asked to reference to sample outputs. Does that mean there is no outputs/print outs?

A: No. Always refer to the webpage and the sample outputs in sample.zip for the standard outputs required.



Q: What does it mean by two transaction being the same?

A: By 3 criteria: i) if they have same amount? ii) if they are both (un)realized? iii) if they are on same date?



Q: What is the use of 'const int MAX_NUM_TRANS = 100;' in ledger?

A: It has been removed to avoid confusion with maxNumTrans. As a consequence, in Account::Account(), "Maximum #. transactions allowed for each ledger: " and "Current #. transactions allowed for each ledger: " will use the same variable for print outs.



Q: Why does ZINC shows red highlights when doing output compares even the lines are the same?

A: If you see a green tick it means you pass the test case; if you see a red cross it means you fail the test case. In the former case, the red highlights just indicate spacing and newline differences, which would be ignored during grading. However, if that is latter case, somewhere in your output does not match that of the standard one.



Q: Why I get 0/0 in ZINC?

A: If you see a green tick it means you pass the test case; Since the testing are not graded, they are of zero mark.



Q: Can I use other libraries?

A: Please refer to the list of allowed libraries here.



Q: Do you provide mapping between print outs to function?

A: Yes. It can be found in toPrint.zip in skeleton.zip.



Q: when should I print "Fail to add the transaction! >.<" in void Account::addTransactionToLedger(Transaction* newTransact) ?

A: It is used to give you to check if your casting has errors. Thus, you cannot see this output if your implementation is correct. It helps you with code checking. Thus, we are not having tests on this error message requiring you to print this.



Q: when should I print "Invalid transaction number!"?

A: When using wrong numTransact in non-empty ledger.



Q: What are the differences between the following print outs?

A: "Sorry, no transaction to print!": when ledger is empty;
"Invalid transaction number!": when ledger non-empty but get wrong transaction number
""Wrong transaction number! Cannot retrieve transactions!": when ledger non-empty but get wrong transaction number



Q: If nTrans > currNumTrans in Ledger::printRecentNTrans(int nTrans) const, do we need to print all the transactions? (suppose currNumTrans > 0)

A: No. Print nothing. And you need to ensure the program should not crash as well.



Q: What to return when the numTransact is <0 in Account::removeTransactionFromLedger(Ledger* led, int numTransact)?

A: It will never happen.



Q: Will ledger constructed with currNumTrans greater than 0?

A: No. Ledger initialization will only has 0 currNumTrans.



Q: Use 'Expenses' or 'Expense' or 'expenses' or 'expense'?

A: Please stick to ZINC.



Q: How to decide if I need to deallocate the memory for an object in a function or not?

A: Referring to the main cpp files, if an object is not deleted in main cpp files, you need to handle the deallocation in your program. There are either two places: either deallocate in function or in destructors. You may refer to sample outputs/ZINC for hints as of where to deallocate them.



Q: So we will still update the balance when the ledger is full in addTransactionToLedger()?

A: Yes.



Q: In Account::updateLedger, if old trans has the same type as new trans and old trans is not found in ledger right, should we update the balance?

A: Yes. There can EXIST such case.



Q: Is it good enough if our program can cover all the sample output, will there be any hidden cases that the sample output does not cover when it comes to grading?

A: Good enough. Similar to pa1, test cases are a bunch of mains cpp which try to test each of your functions separately. So if you pass pre-deadline test cases, you are pretty much ready for hidden tests. But we cannot guarantee you will get full marks just based on that. It is impossible to include all possible cases in pre-deadline tests/sample outputs/ZINC. We assume if you pass pre-deadlines, your program's logic should be correct.
