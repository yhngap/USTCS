Objectives & Intended Learning Outcomes

The objective of this assignment is to provide you with practice on the OOP concepts of Inheritance and Polymorphism, as well as writing code with less reliance on a provided skeleton code.

Upon completion of this assignment, you should be able to:
  1. Write appropriate header files on your own, without being provided them as part of the skeleton code.
  2. Manage dynamic memory effectively with the use of class Constructors and Destructors.
  3. Use Inheritance to allow for significant code reuse between Base classes and Derived classes.
  4. Use function overriding in Derived classes to provide different function implementations for the same member function prototype.
  5. Understand how Base class pointers/references can point/refer to Derived class objects, yet polymorphically call Derived class functions that are declared virtual.
  6. Code a program following the OOP paradigm.
End of Objectives & Intended Learning Outcomes

Introduction

In this programming assignment, we will be implementing a Mini Turn-Based Strategy Game with Console UI, inspired by games such as Wargroove and the Langrisser series. Players (2 or more) take turns to command their Units in battle on a grid-based coordinate system. The winner is the last Player with unit(s) remaining. It is possible for there to be no winner, if all remaining units are wiped out in a single turn of battle.

System Overview

  • PA3_main merely creates a GameEngine instance and invokes it to start the game.
  • GameEngine contains the actual "main" portion of the program, and handles most of the gameplay and Console UI code.
  • GameMap handles the grid state and grid display to Console UI.
  • Player represents a Player in the game, with a set of Units to command.
  • Unit is the base class that represents a combat unit in the game. All the different kinds of units inherit from this base class.

GameMap & Grid-Based Coordinate System

The grid-based coordinate system is a simple (row, column) system with the origin (0, 0) at the top-left corner. (1, 0) is one cell below the origin, while (0, 1) is one cell to the right of the origin. Distance in this game is calculated as Manhattan distance, which is |destination_row - starting_row| + |destination_col - starting_col|. E.g. the distance between (5, 6) and (10, 3) is 8. Diagonal traversal doesn't exist in this grid-based coordinate system.

The game map is stored in two mirrored dynamic arrays:
  • terrain_map stores internal terrain state information, such as terrain obstacles, occupied by units, or empty.
  • display_map stores the actual display characters to be printed on the console.

For the sake of simplicity, we only have one type of terrain obstacle, represented by the '#' character, which is completely impassable. Units are each represented by a single character, which also serves as the Unit ID. Empty terrain is represented by a single blank space character.

The game map is loaded from a textfile. The following is an example:

The first two numbers represent the number of rows and columns respectively in this game map. The following lines then specify the actual game map with '#' and space characters. For the sake of simplicity, no error-checking nor input-sanitization is performed on the game map input textfile.


Players & Units

Players have a name (stored as C++ string) and a set of Units (stored as a dynamic array).

Units have:
  • char id: Used to uniquely identify units, and also for display on the game map. This also limits the number of units possible in a game.
  • UnitType unit_type: A scoped enum. Unit Types will be further elaborated in the Unit Affinity System section.
  • int health: A unit is defeated/dead if its health drops to 0.
  • int attack: Represents the unit's damage potential.
  • int defense: Represents the unit's resistance to damage.
  • int attack_range: Represents the distance from enemy units that a unit can attack.
  • int movement_range: Represents the distance that a unit can move in a single turn.
  • int position_row, position_col: Represents a unit's current position on the game map.
  • bool ready_state: Used to keep track if a unit has already used up its turn.
The players and units are also loaded from a textfile. The following is an example: 
2
Red 4
A Swordsman 4 7
B Swordsman 5 6
C Knight 3 6
D Knight 4 5
Blue 3
X Pikeman 3 21
Y Pikeman 4 22
Z Archer 11 22

The first number represents the number of players. The following string and number represents the player's name and number of units respectively. Finally, each unit begins with its character ID, followed by the Unit Type, then its starting location (row then column). Likewise, no error-checking nor input-sanitization is performed on the players and units input textfile.


Turn-Based Combat

Within a player's turn, he/she can issue commands to all of his/her units. This provides a strategic advantage for having a larger number of units.

Each of player's unit within the turn can choose to perform two sets of actions: Move & Attack, or Heal:
  • Move & Attack
    • Move a distance upto the movement range of the unit. Unused movement range is NOT carried over to the next turn.
    • After Moving, a unit can choose to subsequently Attack a single Enemy Unit within its attack range.
    • A unit can choose to directly Attack without first Moving, but it cannot then Move afterwards (i.e. no Hit-and-Run).
  • Heal: Restores health to the unit, but that unit cannot Move nor Attack for the rest of that turn.

When Unit A attacks Unit B, if Unit A is also within the attack range of Unit B, then both Units will attack each other simultaneously - damage is calculated for both units at the same time with their current stats at the moment the attack is confirmed, and damage is applied to both units at the same time after the attack ends.

In other words, no matter who starts the attack, the defender will automatically counterattack at the same time, as long as the attacker is within range of the defender. Thus, ranged units have a strategic advantage when attacking melee units, as the ranged units will be out-of-range of the melee units. On the other hand, it is risky for weaker units to attack stronger units, as the weaker unit may end up being defeated in the process.

Damage is calculated based on a simple formula of { (A.attack - B.defense) * (A.health / A.max_health) * Unit_Affinity_Modifier } rounded (NOT casted) to an integer.


Unit Affinity System

For our simple game, we only have 4 UnitTypes. Each Unit Type has strength(s) and weakness(es) towards other Unit Type(s), which are represented as a simple modifier to the damage formula.

  • Sword: 1.5 to Spear, 0.5 to Horse.
  • Spear: 1.5 to Horse, 0.5 to Sword.
  • Horse: 1.5 to Sword, 0.5 to Spear.
  • Arrow: No modifiers.

The 4 Unit Types correspond to the 4 Units in our game:
  • Swordsman: Sword Type.
  • Pikeman: Spear Type.
  • Knight: Horse Type.
  • Archer: Arrow Type.
The stats for our 4 Units are as follows:
  • Swordsman: Health 10, Attack 10, Defense 5, Attack Range 1, Movement Range 5.
  • Pikeman: Health 10, Attack 8, Defense 7, Attack Range 1, Movement Range 3.
  • Knight: Health 10, Attack 14, Defense 3, Attack Range 1, Movement Range 10.
  • Archer: Health 10, Attack 9, Defense 1, Attack Range 5, Movement Range 3.

A more complex game would have multiple units of different stats but with shared Unit Type (e.g. Light Cavalry and Armored Cavalry can both be Horse Type and differ on defense and movement range).

Task Description

This time the skeleton code is incomplete and cannot be immediately compiled, in order to provide practice in creating and coding source files from scratch.


GameEngine

Almost everything is given to you in the skeleton code. The entry point is the run_game() function. Recommend reading through all the other functions and control flow in order to get an overview of the whole program.

Only the bool is_game_over() const function definition is missing. It should return true when there is only 1 Player with alive units remaining, or when all Players and Units are wiped out.


GameMap

Similarly, almost everything is given to you in the skeleton code, and recommend reading through it.

Only two function definitions are missing:
  • void update_terrain_map(int row, int col, TerrainState state) is straightforward.
  • bool is_valid_path(int starting_row, int starting_col, int delta_row, int delta_col, int movement_range) has 4 considerations:
    • GameMap Out-of-Bounds.
    • Terrain Blocked (the '#' character).
    • Terrain Occupied by another Unit.
    • Outside of movement range.

For the sake of simplicity, we assume that Units can "teleport" past obstacles and other Units, so we don't need to do any actual pathway calculations.


Player

Only the header file is given in the skeleton code. Create and code the Player.cpp source file accordingly. The required function definitions are all straightforward, based on the descriptive function names, parameter names, and return value.


Unit

Only the to_string_base() function is given, so you don't have to worry about the Console I/O tedium.
Most of the other functions are similarly straightforward. Only special cases will be listed here.

  • virtual void heal() - Restore health equal to the static const int HEAL_RATE variable. Remember to cap according to the static const int MAX_HEALTH variable, and not overflow.
  • virtual void move_delta(int delta_row, int delta_col) - In terms of relative change to row and column position, instead of the absolute destination, so it is more convenient when inputting Move commands.
  • virtual void attack_unit(Unit* const defender) - Keep in mind that the defender also "fights back" at the same time if the attacker is inside the defender's attack range.
  • void receive_damage_actual(double actual_damage) - Helper function. You can place code that is the same across all Units here. Remember not to go below 0 health.
Note that the function declarations with = 0 signify that they are pure virtual, i.e. have no function definition, thus are expected to and must be overriden in the derived classes.


Derived Unit Classes

No skeleton code provided. The topic of this assignment is OOP Inheritance and Polymorphism, and coding the derived class declarations is part of the fundamentals.

  • virtual void receive_damage_affinity(double raw_damage, UnitType attacker_unit_type) - Refer to the Unit Affinity section in the Introduction.
  • virtual string to_string() const - The derived Unit classes should append their class name to the end of the base string. E.g. Archer should append "Archer".


Additional Remarks

  • You are allowed to use the <cmath> library for the rounding and absolute-value math functions, which is already included in the skeleton code.
  • You are allowed to, but discouraged from, using the C/C++ standard libraries. However, you are NOT allowed to use external libraries.
  • You are allowed to define your own helper code (functions, structs, classes, etc), but be careful not to crash ZINC.
  • Remember to properly deallocate (delete) dynamic memory in the class Destructors (if applicable).
  • Apologies in advance again for the ugly Console UI and tedious usage.

Frequently Asked Questions

This section will be expanded as your fellow classmates ask more questions about this programming assignment. It is in your interest to check this section regularly for updates and clarifications even if you have already submitted your work by then.

Q: The demo program enters an infinite loop when given unexpected input (e.g. inputting a character when expecting an integer). Is this a bug?
A: This is just the behavior of cin >> variable; when given input that is not type-matched. You don't need to worry about these user-input-error cases for PA3, as all console I/O is already handled for you by the skeleton code. ZINC will perform the final grading with GTest, similar to PA1.

Q: What are the restrictions regarding modifying the header files, including extra header files, etc?
A: The only hard restriction is that your code should compile and not crash on ZINC, though this is discouraged.

Q: Can you give an example of what "simultaneous" means for the Turn-Based Combat?
A: Please have a look at this YouTube video. Regardless of who initiates the attack, as long as the attacker is not out-of-range of the defender, then both units will enter combat simultaneously. It is NOT attacker deals damage then defender deals damage.

Q: Why does ZINC compiler show this warning for the function Unit* const* const get_units() const in Player class? "warning: type qualifiers ignored on function return type [-Wignored-qualifiers]"
A: That warning just means the underlined const in Unit* const* const get_units() const is redundant. get_units() is returning-by-value a pointer-to-a-pointer Unit**, so Unit** const func() is similarly redundant as a return-by-value const int func(). No changes in the behavior of the code, which is why it is only a warning and not an error.

End of FAQ
