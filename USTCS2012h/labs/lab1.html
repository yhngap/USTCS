Introduction

This is a lab about program flow control: Implement a simple rock-paper-scissors-based game against a computer-controlled opponent.


How to Play?

  • The Player and the Computer both start with maximum HEALTH and no CHARGE.
  • In each round, the Player and the Computer simultaneously choose one of the following actions:
    • ATTACK:
      • Decreases opponent's HEALTH with DAMAGE depending on one's CHARGE.
      • Consumes all of one's CHARGE if any (reset to 0).
      • Interrupts if opponent is CHARGING (reset opponent's CHARGE to 0).
    • DEFEND:
      • Reduces DAMAGE from opponent's ATTACK.
      • Possibly NULLIFYING or even REFLECTING DAMAGE.
      • Protects against being interrupted and preserves CHARGE.
    • CHARGE:
      • Increases one's CHARGE by 1, empowering a future ATTACK.
      • Can be interrupted if the opponent ATTACKS at the same time.
      • CHARGE count is preserved when DEFENDING.
  • The game ends when someone's HEALTH reaches 0, and the winner is the one with non-zero HEALTH. If both the Player's and the Computer's HEALTH reach 0 simultaneously, the game ends in a tie.

Letting the Computer Play

In order to keep the game dynamic and unpredictable, the computer opponent will choose its actions randomly. You need not to worry on coding this as it is already done for you in the skeleton code. However, it is useful for you to be aware of random number generators.

Random number generators (RNGs) generate sequences of numbers which have the desired properties of randomness. Since generating truly random numbers is relatively costly, a class of RNGs called pseudo-random number generators (PRNGs) output numbers which have the desired properties without being truly random. Typically, PRNGs are composed of an internal state and an algorithm which generates the next pseudo-random number according to the internal state and which also updates the current internal state. Hence, the non-deterministic property of a PRNG mainly relies on not knowing its internal state and its algorithm. On the other hand, if the internal state can be set, we can return to a previous internal state in order to repeat the sequence of numbers generated. This property is useful for replicating experiments with PRNGs, and thus, most technical reports include the random seed which is the number programmers used to set the initial internal state.

For our lab, it is easier to debug and also to perform automated grading if we can fix the seed and know the pseudo-random sequence beforehand. For other use cases where this is undesirable, usually an easy solution is to use the current time as the seed, which is always incrementing (and thus always changing). You may have rarely received an error with HKUST's Duo 2FA system with the message of "failed to synchronize with time server" or something similar. More advanced random number generation systems are typically used in security and cryptography applications, such as financial authentication systems.

For simple use cases, usually the PRNG used in C++ is the srand() and rand() functions in the cstdlib library, which must be imported or included at the beginning of the program. The following is a code example:

#include <iostream> // For cout
#include <cstdlib>  // For srand, rand
#include <ctime>    // For time
using namespace std;

int main() {
  /*
     srand is a function used to initialize the pseudo-random number generator.
     An unsigned int is a parameter being used to generate a sequence of integers.
     We call it a seed.

     In the following statement,
     - time(nullptr) returns the current time of the system in number of seconds elapsed
       since Jan 1, 1970 and is used as the seed to generate the sequence.
  */
  srand(static_cast<unsigned int>(time(nullptr)));

  /*
     In each iteration, get the next number in the sequence, make it in the range 0 to 9
     and print it on screen.
  */
  for(int i=0; i<10; ++i) {
    cout << rand() % 10 << ( (i != 9) ? " " : "\n" );
  }

  // Question: Check the following code and see what you observe.

  srand(100);
  for(int i=0; i<5; ++i) {
    cout << rand() % 10 << ( (i != 4) ? " " : "\n" );
  }

  srand(100);
  for(int i=0; i<5; ++i) {
    cout << rand() % 10 << ( (i != 4) ? " " : "\n" );
  }

  return 0;
}

However, for this lab we are using our own custom PRNG implementation, in order to ensure consistency for automated grading. Please do not modify the PRNG code blocks in the skeleton code, otherwise you may break the automated grading pipeline and result in a false "0" grade.

The "Choose Computer Player (0-100): " portion of the code asks the player for an integer input between 0-100 to be the seed of our PRNG. As explained above, reusing the same seed will produce the same sequence of pseudo-random numbers across multiple runs of the game.

Here is what the code looks like, though it is unnecessary for you to understand it when coding the lab tasks. 

// Our own simple implementation of a pseudo-random number generator, for debugging and automated grading consistency.
static unsigned long rng_lab1 = 0;

static void _srand(unsigned seed) {
	rng_lab1 = seed;
}

static int _rand() {
	rng_lab1 = (1103515245 * rng_lab1 + 12345) % (1 << 31);
    return rng_lab1 & 0x7FFFFFFF;
}

int main() {
	// Welcoming Message.
	cout << "Welcome to COMP 2012H Lab 1!" << '\n';
	cout << "ATTACK, DEFEND, CHARGE!" << '\n';
	cout << endl;
	
	// Setting Computer Player.
	cout << "Choose Computer Player (0-100): ";
	unsigned int id;
	cin >> id;
	_srand(id + 2012); // Initialize pseudo-random number generator.

}

Lab Work

Please refer to the Demo Program and Sample I/O for the appropriate output string formatting. A large comment block of Output Messages has been included in the skeleton code for your copy-paste convenience.


Task 1 - Handle Player's ACTION and Computer's ACTION

Ask the Player to choose between ATTACK(0), DEFEND(1), and CHARGE(2).
If the Player chooses CHARGE while at FULL CHARGE, notify the Player and ask again.
If the Player's input is invalid, notify the Player and ask again.
Announce the Player's action, and afterwards, announce the Computer's action.
Note: You can safely ignore the situation of wrong data type cin input (it will crash the program).


Task 2 - Determine Player's DAMAGE and Computer's DAMAGE

Determine player_damage and computer_damage according to the provided chart.
Announce REFLECTED DAMAGE and NULLIFIED DAMAGE whenever they occur.

ATTACK's
CHARGE		 ATTACK's DAMAGE
without DEFEND with DEFEND
0 1 -1*
1 2 0**
2 5 2
3 10 5

*REFLECTED DAMAGE **NULLIFIED DAMAGE


Task 3 - Update Player's CHARGE and Computer's CHARGE

Update player_charge and computer_charge according to the following rules:

  • If one ATTACKS, one loses all CHARGE.
  • If one CHARGES and the opponent ATTACKS, one's CHARGING is INTERRUPTED and one's CHARGE resets.
  • If one CHARGES and the opponent does not ATTACK, one's CHARGE increments.
Announce INTERRUPTED CHARGING whenever it occurs.


Task 4 - Update Player's HEALTH and Computer's HEALTH

Update player_damage and computer_damage.
Cap DAMAGE received to current HEALTH (no overkill).
Announce player_damage (only if non-zero) and update player_health.
Similarly, announce computer_damage (only if non-zero) and update computer_health.


Task 5 - Check for winner

Set winner to 1 if the Player wins, 2 if the Computer wins, and 3 for a tie.
One wins if the opponent's HEALTH reaches 0.
A tie occurs if both HEALTHs simultaneously reach 0.


Task 6 - Display Ending Message

Output appropriate ending message according to winner.

End of Lab Work
